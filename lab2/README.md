# Лабораторная работа №2

## Тема

Сравнение алгоритмов поиска: бинарный поиск, сбалансированное двоичное
дерево поиска (AVL) и хэш-таблица с разрешением коллизий методом цепочек.

## Цель работы

Реализовать три принципиально разных подхода к ассоциативному доступу,
сравнить их асимптотику и **экспериментально** замерить время поиска на
больших случайных наборах данных.

## Реализованные структуры

### Бинарный поиск

```cpp
template<typename T, typename Key>
int BinarySearch(const std::vector<T>& data, Key key,
                 std::function<Key(const T&)> getter);
```

Работает с *уже отсортированным* массивом. `getter` позволяет искать
по произвольному полю объекта. Содержит защиту от пустого массива
(иначе `data.size() - 1` на беззнаковом типе уходит в переполнение).

| Операция | Сложность |
|----------|-----------|
| Find     | `O(log n)` |
| Подготовка (сортировка) | `O(n log n)` |

### Бинарное дерево поиска (AVL)

Самобалансирующееся дерево. Наивный BST при вставке **отсортированной**
последовательности ключей вырождается в связный список высотой `n`,
и поиск деградирует до `O(n)`. AVL хранит в каждом узле высоту и после
каждой вставки восстанавливает баланс поворотами (случаи LL, RR, LR, RL),
поэтому высота гарантированно остаётся `O(log n)` на любых данных,
включая уже отсортированные.

| Операция | Любой случай (включая отсортированный вход) |
|----------|---------------------------------------------|
| Insert   | `O(log n)` |
| Find     | `O(log n)` |

Дополнительно: `Height()` и `Count()` (используются в тестах для проверки
балансировки), деструктор для освобождения узлов, обновление значения при
повторной вставке существующего ключа. Возврат «не найдено» закодирован
значением `-1` (подходит для значений-индексов; для произвольного `Value`
стоило бы перейти на `std::optional`).

### Хэш-таблица

Разрешение коллизий методом **цепочек** (chaining): каждая ячейка —
`std::list` из пар `(Key, Value)`. Хэш-функция — `std::hash<Key>() % capacity`.
В интерактивном режиме `capacity` задаётся равной размеру набора, чтобы
держать коэффициент заполнения около 1.

| Операция | Среднее | Худший случай |
|----------|---------|---------------|
| Insert   | `O(1)`  | `O(n)` (все ключи в одну корзину) |
| Find     | `O(1)`  | `O(n)` |

## Интерактивный режим (`search_app`)

По аналогии с lab1 — меню в цикле:

```
1 - Generate random dataset       # N случайных пар (key, value), сортировка по ключу
2 - Compare search efficiency     # M случайных запросов через все три структуры
3 - Find key in all structures    # поиск одного ключа всеми способами
0 - Exit
```

Сравнение строит все три структуры, прогоняет `M` случайных запросов
(половина — заведомо существующие ключи, половина — произвольные) и печатает
время построения и поиска в микросекундах, а также высоту AVL-дерева. Все три
структуры находят одинаковое число ключей — это служит проверкой корректности.

Пример вывода (200 000 записей, 100 000 запросов):

```
Dataset size: 200000, queries: 100000
AVL tree height: 18  (degenerate BST would be 200000)
-----------------------------------------------------------
Binary search : lookup 85941 us  (found 54697)
AVL tree      : build 148178 us, lookup 46527 us  (found 54697)
Hash table    : build 98090 us, lookup 36145 us  (found 54697)
```

Видно главное: высота дерева — 18 (≈ log₂ 200000), а не 200000, то есть
балансировка работает; хэш-таблица быстрее всех на поиске.

## Тесты (`search_tests`)

Проверяются **все три** метода:

- `TestBinarySearch` — края/середина массива, отсутствующий ключ, пустой массив;
- `TestBinaryTree` — поиск, отсутствующий ключ, обновление значения, отсутствие
  дубликатов по ключу;
- `TestBinaryTreeBalanced` — **ключевой тест**: вставка 1023 строго возрастающих
  ключей; проверяется, что все находятся и высота `<= 14` (то есть дерево не
  выродилось в список — наивный BST дал бы высоту 1023);
- `TestHashTable` — поиск и отсутствующий ключ.

## Нюансы

- **`assert` и шаблонные запятые.** `assert(BinarySearch<int, int>(...))` не
  компилируется: `assert` — макрос, и запятая в `<int, int>` рвёт список его
  аргументов. Лечится внешними скобками: `assert((...))`.
- **Сортировка перед бинарным поиском обязательна** — набор формируется уже
  отсортированным по ключу.
- **Именно отсортированный вход** мотивирует переход с BST на AVL: данные для
  бинарного поиска и так сортируются, и наивное дерево на них деградировало бы.

## Сборка и запуск

```bash
cmake -S . -B build
cmake --build build

./build/search_app    # интерактивный режим: генерация и сравнение
./build/search_tests  # автотесты всех трёх структур
```

> На Windows с MSVC бинарники лежат в `build/Debug/` (`search_app.exe`),
> на Linux/Docker и при генераторе Ninja — прямо в `build/`.

В Docker:

```bash
docker build -t lab2 .
docker run --rm lab2          # CMD = прогон тестов

# интерактивный режим в контейнере:
docker run --rm -it lab2 ./build/search_app
```
