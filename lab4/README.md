# Лабораторная работа №4

## Тема

Применение OpenMP к итерационному методу релаксации Якоби в
трёхмерной кубической сетке. Сравнение последовательной и параллельной
реализаций по времени и ускорению.

## Постановка задачи

В кубе `N × N × N` точки на границе зануляются, во внутренних узлах
задаётся начальное распределение `A[i][j][k] = 4 + i + j + k`.
На каждой итерации выполняются две фазы:

1. **relax** — для каждой внутренней точки вычисляется новое значение
   `B[i][j][k]` как среднее шести соседей по массиву `A`.
2. **resid** — копируем `B` в `A` и одновременно ищем максимальную
   по модулю разницу `eps = max |A - B|`.

Цикл идёт до достижения `itmax` итераций или пока `eps < maxeps`
(по умолчанию `1e-8`).

## Особенности и нюансы

- **Линейный layout 3D-массива.** Вместо `vector<vector<vector<double>>>`
  используется один `std::vector<double>` с индексацией
  `Idx(i,j,k) = (i*N + j)*N + k`. Это важно для производительности:
  данные лежат непрерывно, kernel `relax` хорошо предсказывается
  prefetch-механизмом процессора.
- **Параллелится только внешний цикл по `i`** (`#pragma omp parallel for
  schedule(static)`). Соседи `i-1` и `i+1` читаются, но не модифицируются
  на той же итерации — гонок нет, потому что запись идёт в массив `B`,
  а чтение — из `A`.
- **Reduction вручную в `ResidParallel`.** Вместо `reduction(max:eps)`
  (требует OpenMP 3.1+ и не всегда дружелюбен к компилятору) используется
  паттерн «локальный max → critical». Каждый поток считает `localEps`,
  затем под `#pragma omp critical` обновляет общий `eps`. Это
  эквивалентно reduction, но переносимо.
- **Контрольная сумма для верификации.** `Verify()` вычисляет
  взвешенную сумму всех точек куба и используется в тестах для
  проверки, что параллельная версия даёт *тот же* результат, что
  и последовательная.
- **Release-сборка по умолчанию.** В `CMakeLists.txt` явно
  выставлен `CMAKE_BUILD_TYPE=Release`, если он не задан.
  Без оптимизаций (`-O0`) разница между serial/parallel «съедается»
  накладными расходами OpenMP — это ключевой момент при анализе
  замеров.

## Запуск

```bash
cmake -S . -B build
cmake --build build

# Последовательная версия
./build/jacobi_app serial 128 12

# Параллельная версия (2 потока)
./build/jacobi_app parallel 128 12 2
```

```bash
docker build -t lab4 .
```

## Автоматическое построение таблицы и графиков

Чтобы не заполнять таблицу вручную, есть Python-скрипт
`analysis/plot_results.py`. Он запускает `jacobi_app table <k> <csv>`,
читает выгруженный CSV и генерирует в директорию `results/`:

- `results.csv` — сырые замеры;
- `results.md` — та же таблица в формате Markdown;
- `time_vs_n.png` — график времени работы от размера сетки `N`;
- `speedup_vs_n.png` — график ускорения от размера сетки `N`.

```bash
# Один раз поставить зависимости
pip install -r analysis/requirements.txt

# Собрать проект (нужен jacobi_app), затем построить таблицу и графики
cmake --build build
python analysis/plot_results.py -k 10

# Построить только графики по уже готовому CSV, не запуская бенчмарк
python analysis/plot_results.py --no-run --csv results/results.csv
```

Полезные ключи: `-k` — вариант по списку, `--binary` — путь к
`jacobi_app`, `--out` — директория результатов, `--no-run` — строить по
существующему CSV.

## Закон Амдала

Если доля последовательного кода `s`, а параллельного `1 - s`,
то теоретическое ускорение на `p` ядрах ограничено

```
S(p) ≤ 1 / (s + (1 - s) / p).
```

На практике `relax` и `resid` хорошо распараллеливаются по `i, j`,
и наблюдаемое ускорение на 2 ядрах близко к 2, но не превышает
теоретического предела из-за накладных расходов на создание потоков
и синхронизацию reduction. На размерах `N < 64` параллельная версия
может оказаться **медленнее** последовательной — стоимость fork/join
превышает выигрыш.

## Тесты

`jacobi_tests` проверяет:

- инициализация выдаёт ненулевую контрольную сумму;
- одна итерация даёт положительный `eps`;
- параллельная версия даёт ту же контрольную сумму, что
  и последовательная (`|Δ| ≤ 1e-9`).
