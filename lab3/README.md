# Лабораторная работа №3

## Тема

Введение в OpenMP. Знакомство с базовыми директивами на пяти простых
примерах из методички.

## Реализованные примеры

| № | Файл                              | Демонстрирует |
|---|-----------------------------------|---------------|
| 1.1 | `example1_hello.cpp`            | `#pragma omp parallel`, `omp_get_thread_num`, `omp_get_num_threads` |
| 1.2 | `example2_dotproduct_for.cpp`   | `#pragma omp parallel for reduction(+:sum)` |
| 1.3 | `example3_dotproduct_subroutine.cpp` | вложенные `parallel` + `for reduction` в подпрограмме |
| 1.4 | `example4_dotproduct_sections.cpp` | `#pragma omp sections` (явное разбиение работы на две секции) |
| 1.5 | `example5_matrix_vector.cpp`    | `#pragma omp for` + `#pragma omp critical` для согласованной печати |

## Особенности и нюансы

- **`omp_set_num_threads` вызывается перед `#pragma omp parallel`.**
  Если число потоков, переданное аргументом, > 0 — оно явно
  устанавливается; иначе используется значение по умолчанию
  (`OMP_NUM_THREADS` или количество логических ядер).
- **Reduction-переменная.** В `Example 1.2` `sum` находится в `reduction(+:sum)`:
  у каждого потока создаётся приватная копия, по выходе из параллельного
  региона все приватные значения суммируются в общую. Без `reduction`
  получили бы гонку данных.
- **Глобальные переменные в Example 1.3.** Чтобы продемонстрировать
  «подпрограмму, разделяющую состояние с потоками», `g_sum`, `g_a`, `g_b`
  объявлены в анонимном namespace. Это намеренный пример из методички;
  в продакшен-коде передавали бы параметры явно.
- **`sections nowait`** в Example 1.4 — два цикла исполняются параллельно,
  и потоки не ждут друг друга на конце блока `sections`. Финальная
  синхронизация всё равно происходит на закрывающей скобке `parallel`.
- **`critical` в Example 1.5** используется именно для печати, а не для
  вычислений. Без него `printf` от разных потоков смешивал бы вывод;
  на сам результат вычисления `c[i]` это не влияет — там нет гонки,
  потому что каждый поток пишет в свой `i`.
- **Все примеры детерминированы.** Входные данные заполняются
  по правилам `a[i] = b[i] = i`, что упрощает тестирование: ожидаемая
  сумма `sum_{i=0}^{n-1} i² = n(n-1)(2n-1)/6` известна заранее.

## Сборка и запуск

```bash
cmake -S . -B build
cmake --build build

./build/openmp_intro_app 2     # запуск с 2 потоками, интерактивное меню
./build/openmp_intro_tests     # автотесты корректности
```

В Docker:

```bash
docker build -t lab3 .
docker run --rm lab3
```

## Тесты

`openmp_intro_tests` проверяет, что значения, полученные параллельными
версиями примеров (скалярное произведение, поэлементное сложение,
матрично-векторное произведение), совпадают с эталонными — посчитанными
последовательно.
